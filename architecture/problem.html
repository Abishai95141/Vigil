00 · v3 §2 Problem statement

Kubernetes is observable. It isn't interpreted.

Telemetry is abundant. The catalogue of every signal, tool and gotcha exists. What doesn't exist is a single operator with time to correlate it under incident pressure — especially in the places where Kubernetes runs but SREs don't.

The questions that motivate v3
No dashboard answers these.
Each one requires correlation across two or more resource types. None of them are exotic — they're what every on-call asks first.
Q1
"Which pod is causing the CPU spikes?"
pod ↔ container ↔ cgroup ↔ node
Q2
"How are PVC I/O patterns linked to pod restarts?"
pvc ↔ pod ↔ kubelet evictions ↔ events
Q3
"Are services influencing each other's resource use?"
service ↔ pod-role ↔ node ↔ neighbour pods
